01 The Product

One dashboard. The entire job hunt.

02 The Problem

The graduate job hunt is a data problem wearing a motivation problem's clothes.

Hunting for a role in Dublin as an international MSc candidate means a dozen portals, hundreds of listings, ATS filters that reject silently, and the same CV-tailoring ritual repeated nightly. The hours go to admin; the minutes left over go to strategy — exactly backwards.

I treated it the way I'd treat any operations problem from my data-science years: instrument it, automate the collection, score the options, and spend human attention only where it changes the outcome.

03 What It Does

An agent that hunts while I sleep.

  • Daily automated scans. A scheduled Claude agent sweeps the market every morning for roles matching my profile — AI governance, data, analyst and graduate tracks — and files what it finds into the dashboard.
  • Deep role-fit analysis. Each promising role gets a structured report: fit score, role-fit narrative, ATS readiness, keyword gaps to address, and tailored interview-prep questions.
  • A live pipeline. Applications grouped by location and stage, with expired-listing handling, internship filters, experience-level filters, and manual add-a-job for roles found in the wild.
  • Keyword intelligence. Aggregated across every analysed role: which skills the market keeps asking for that my CV under-sells — feeding directly back into how I position myself.
04 Built With Claude

Claude is the engine — and the co-engineer.

JobPilot runs on scheduled Claude agent tasks: the same agent that helped design and build the dashboard executes the daily scans and writes each analysis as structured JSON the dashboard renders. Designing those schemas — what a "fit score" must contain to be trustworthy, how a gap report should be phrased to be actionable — was prompt engineering as product design.

It also gave me a genuinely production-flavoured incident: a new analysis schema ("Deep Analysis v1") started landing alongside the old format, and the dashboard — which assumed one shape — crashed on render. The fix was a normalisation layer at the data boundary that converts every schema version into one canonical shape, unit-tested against all 40+ real analyses. Schema versioning, data contracts, graceful degradation: the exact disciplines AI governance frameworks ask of production ML systems, learned on my own stack first.

"The best way to understand what governing an AI system takes is to be the one who has to trust its outputs every morning."

07 What I Learned

Automation without observability is just faster chaos.

Key Takeaways
  • Normalise at the boundary. When an AI writes your data, schema drift isn't an edge case — it's a certainty. One canonical shape, enforced at ingestion, saved the whole system.
  • An agent is a colleague: brief it well. The quality of every analysis traced back to how precisely the task and output contract were specified. Vague prompts produce confident nonsense.
  • Trust needs receipts. I only rely on JobPilot's scores because I can audit how they're produced — the same standard the EU AI Act sets for systems that score other people.

Privacy by design: everything — CV, analyses, pipeline — lives locally. No third-party trackers, no data leaving the machine. Art. 5(1)(c) isn't just something I study.
